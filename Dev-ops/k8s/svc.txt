
==================================
Service:
==================================
 - Services are used to expose the Pods to outside of the cluster.
 - There are 3 types of services:
	1) Cluster IP:
		- All the Pods will be exposed within the cluster (internally).
		- They will have single IP address.
		- Cluster IP service is used for internal communication.
		- It is default service.
	2) Node Port:
		- Expose the pods outside cluster.
		- Access Pods using Node public IP and port number.
		- port range -> 30000 to 32767
		- If we will not assign any port, it will take a random port
	3) Load Balanced:
		- Distribute traffic to Pod replicas.


 - In case of a custom Kubernetes Cluster using minikube or kubeadm, there is no LoadBalancer integrated (unlike AWS or Google Cloud). 
 - With this default setup, we can only use NodePort or an Ingress Controller.

Use case of service:
	Load Balancing
	Service Discovery

Node Port Ex:
======================

Create a deployment: nginx-deploy.yml

apiVersion: apps/v1
kind: Deployment
metadata:
  name: dep-ex1
  labels:
    app-name: nginx-web-app
spec: 
  replicas: 2
  selector:
    matchLabels:
      app-name: nginx-web-app
  template:
    metadata:
      labels:
        app-name: nginx-web-app    ----> service will check the label
    spec:
      containers:
        - name: nginx-cnt
          image: nginx:1.14.2
          ports:
            - containerPort: 80


kubectl apply -f nginx-deploy.yml
kubectl get deploy
kubectl get pods -o wide


Service will look for Pod labels and will forward the request to Pods.
Ex: svc.yml

apiVersion: v1
kind: Service
metadata:
    name: nginx-web-svc
spec:
    type: NodePort
    selector:
        app-name: nginx-web-app
    ports:
      - port: 80
        targetPort: 80
        nodePort: 30008

targetPort - here the application is running


kubectl apply -f svc.yml
kubectl get svc

port - The port on which the Service itself is exposed within the cluster.
targetPort - The port where your application inside the container is actually listening.
nodePort 
	- A static port exposed on each Node's IP address, for external access.
	- Default Value (if omitted) - Kubernetes assigns a random port in the range 30000-32767.




Cluster IP:
========================

Usage:
1. DB Access - Internal applications connecting to DB
2. Micro services - service to service communication
3. Implement cache service
4. Implementing Kafka or RabbitMQ

Ex:

Create a deployment:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app-deployment
  labels:
    app-name: nginx-web-app
spec: 
  replicas: 2
  selector:
    matchLabels:
      app-name: nginx-web-app
  template:
    metadata:
      labels:
        app-name: nginx-web-app
    spec:
      containers:
        - name: nginx-cnt
          image: nginx:1.14.2
          ports:
            - containerPort: 80


Create the service:

apiVersion: v1
kind: Service
metadata:
    name: nginx-web-svc
spec:
    type: ClusterIP
    selector:
        app-name: nginx-web-app
    ports:
      - port: 80
        targetPort: 80

Test using curl to pod using pod ip:

curl http://pod-ip:port
Ex: curl http://192.168.84.132:80

Test using curl to pod using svc:

curl http://svc-ip:port
Ex: curl http://10.111.138.15:80


Explanation:
 - create the pods using deployment.
 - pods will get IP addresses.
 - pods can communicate using these IP addresses.
 - If we delete the pod, IPs will get changed.
 - but, in case of svc, there will be a fixed IP.
 - by using this, pods can communicate.


LoadBalancer:
================
apiVersion: v1
kind: Service
metadata:
    name: nginx-web-svc
spec:
    type: LoadBalancer
    selector:
        app-name: nginx-web-app
    ports:
      - port: 80
        targetPort: 80


kubectl get svc
It will show EXTERNAL-IP as <pending>.

NAME            TYPE           CLUSTER-IP      EXTERNAL-IP   PORT(S)        AGE
kubernetes      ClusterIP      10.96.0.1       <none>        443/TCP        3h47m
nginx-web-svc   LoadBalancer   10.106.52.239   <pending>     80:32012/TCP   12s

LB=LoadBalancer
LB service remains EXTERNAL-IP <pending> state when the cluster's environment lacks a component to automatically provision an external LB and assign an IP.
